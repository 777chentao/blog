    # 8. git版本管理

    # 8.1 git命令

    上传：工作区 => 缓存区 => 本地仓库 => 远程仓库

    工作区 => 缓存区：git add .

    缓存区 => 本地仓库: git commit -m [message]

    本地仓库 => 远程仓库: git push <本地分支名>|<远程分支名>

    拉取：远程仓库 => 本地仓库 => 缓存区 => 工作区

    远程仓库 => 本地仓库: git fetch/clone <仓库地址>

    本地仓库 => 工作区: git checkout <本地分支名> /切换本地分支

    远程仓库 => 工作区: git pull <本地分支名>

    其他：

    查看本地分支: git branch

    查看远程分支: git branch -r

    添加远程版本库: git remote add [仓库名] [url]

    从远程获取/更新代码: git fetch

    # 8.2 管理流程

    1. 建立远程仓库
    2. 代码文件夹cmd中git init初始化
    3. 文件夹中代码提交到缓存区git add .
    4. 提交到本地仓库git commit -m ‘项目初始化’
    5. 关联远程仓库git remote add origin ‘http.......远程仓库url’
    6. 推送远程git push -u origin “远程分支名”

    # 8.3 项目中的使用场景

    需求开发前的分支拉取流程

    1. 从远程分支拉取代码：git pull
    2. 创建本地分支: git checkout -b hyw(分支名)
    3. 代码提交到缓存区: git add .
    4. 提交到本地仓库: git commit -m ‘hyw代码’
    5. 推送到远程,创建hyw远程分支: git remote add origin(远程仓库名) hyw(分支名)

    需求开发后的分支合并流程

    1. 把本地hyw分支合并到本地master中：git merge hyw（此时分支在master上进行合并的）
    2. 将本地master提交远程: git push origin master

    分支冲突解决（可以组件化开发，一个人开发一个组件）

    freeway开发一段代码提交远程后，hyw又开发了一段代码提交远程：

    1. git add .
    2. git commit -m ‘hyw又写了代码’
    3. git push origin hyw (远程hyw分支已更新)
    4. git checkout master (切换到本地master分支)
    5. git merge hyw (合并本地hyw分支到本地master分支，此时有冲突，因为之前freeway提交过一次到本地master分支)
    6. 怎么解决：和freeway沟通问是否需要，若都要选择accept both changes
    7. 在本地master上推送到远程master: git push origin master

    出现线上问题时hotfix分支操作流程

    1. 拉去最新代码
    2. 修改代码后放入本地预上线分支
    3. 推送至远程预上线分支

    # 9. 前后端跨域

    # 9.1 同源策略

    浏览器最核心也最基本的安全功能，如果缺少了同源策略，浏览器很容易受到XSS、CSFR等攻击。

    同源 => 指的是 “协议+域名+端口” 三者的相同 只要有一个不同就会导致跨域问题.

    # 9.2 解决方案

    1. 前端方案（代理服务器）,适用于开发阶段
    // vue.config.js 
devServer: {
        port: 9086,
        open: false,
        overlay: {
            warnings: false,
            errors: true
        },
        proxy: {
            "api": {
                target: "http://11.111.111.11:8000/", => 此处写明自己访问接口的url
                changeOrigin: true,
                pathRewrite: {
                    "^/api": "" 
                    //这里理解成用'/api'代替target里面的地址,比如我要调用'http://40.00.100.100:3002/user/add'，直接写'/api/user/add'即可
                }
            }
        }
    }

// request.js
let instance = axios.create({
    baseURL: "/api",
    withCredentials: true, // send cookies when cross-domain requests
    timeout:5000,
})

    1. 后端方案（CORS）

      实现CORS的关键是服务器，只要服务器实现了CORS接口,就可以跨域通信（在 HTTP 响应头上通过 Access-Control-Allow-(Origin/Headers/Methods/Credentials/) 等响应头字段设置允许访问该资源的请求源，请求头，请求方法，是否允许携带 cookies 等）

      CROS 请求默认不包含 cookie 信息 如果我们想要包含就必须在请求中将 withCredentials 属性 配置为 true

      • 提到简单请求和非简单请求，见 MDN (opens new window) (opens new window)，只有非简单请求才会触发预检过程
      • 浏览器首先发出一个 OPTIONS 请求，包含非简单请求中存在的请求信息，如请求方法，请求头，源等等
      • 服务器端返回它所允许的上述内容，并且如果预检请求能够通过就返回 200 响应
      • 浏览器接收到响应后，再判断是否真正发出非简单请求

    2. jsonp

      改变请求方式 dataType: 'jsonp'

      • 只允许 GET 请求，因此 URL 长度受限，不安全
      • 需要服务器端进行配合

    3. nginx添加请求头

      • nginx 是通过为每一个 HTTP 响应添加上满足 CORS 要求的请求头实现的跨域，本质上请求还是通过浏览器发出的
      • webpack 是通过内部发送请求的方式绕过浏览器获取到结果之后再返回给浏览器，本质上请求不是通过浏览器发出的

    # 10. 登录验证

    # 10.1 传统session身份认证

    1. 浏览器向服务器发送登录请求
    2. 服务器端验证通过，会保存含用户信息的session，生成sessionid
    3. 服务器set-cookie设置cookie:sessionid = xx，并将sessionid返回给浏览器
    4. 浏览器再次发送请求，会在请求头部的cookie中放入sessionid，并同请求数据发送给服务端
    5. 服务器验证session，并根据sessionid查对应用户信息返回结果

    缺点：当服务端为集群时，用户登录其中一台服务器，会将session保存在该服务器的内存中， 但是当用户访问其他服务器时。会无法访问。依赖cookie，存在CSRF安全问题。

    解决方法：采用缓存服务器（redis）保证共享或第三方缓存来保存session

    # 10.2 基于token身份认证

    采用缓存服务器（redis）

    1. 浏览器发送登录请求
    2. 服务器验证通过，生成token，以token为key，将用户信息映射关系存入redis，同时将token返回到浏览器
    3. 将token存入cookie中，再发送带有token的cookie的请求
    4. 服务器拿到token，向redis中获取用户信息，返回结果

    # 10.3 JWT验证（json web token）

    1. 浏览器发送登录请求
    2. 服务器验证通过，生成带用户信息的加密token（jwt），并将jwt返回到浏览器
    3. 浏览器将jwt存入authorization header中，再发送带有jwt的请求头的请求
    4. 服务器拿到jwt，并从jwt中获取用户信息，返回结果

    和11.2不同的是把用户信息存入了token中

    # 10.4 token放在哪？

    1. webstroage

      放在webstorage中，请求时将其放到请求头中。由于webstroage可通过js访问，容易遭受XSS跨站脚本攻击，从而获取到用户信息

    2. cookie

      放在cookie中，请求时自动带上，通过设置httponly可防止js获取cookie，设置secure可只允许https请求时才使用。但是cookie自身的安全性，可能遭受CFRS跨站请求伪造，从而盗用身份信息，恶意操作。

    # 11. XSS

    # 11.1 XSS攻击

    XSS攻击即页面被注入了恶意代码。通常页面包含的用户输入内容都在固定的容器或属性内，以文本形式展示；攻击者利用输入片段，拼接特殊格式的字符串，形成代码片段。攻击者往往在页面内注入脚本，使之在用户浏览器上运行，从而引发潜在风险。

    页面渲染、连接跳转都有可能出现xss攻击

    解决方案：采用html转义、设置白名单

    在处理输入时，以下内容不可信：

    1. 来自用户的UGC信息（用户生成的内容）
    2. 第三方链接
    3. url参数
    4. post参数
    5. Referer
    6. Cookie

    # 11.2 XSS分类

    ​ 存储区 （恶意代码存放位置） 插入点（由谁取得恶意代码，渲染到页面）

    存储型 后端数据库 HTML

    反射型 URL HTML

    dom型 后端数据库/前端存储/url 前端js

    # 11.3 XSS预防

    1. 存储型、反射型是后端责任，dom型前端责任
    2. 转义应在html输出时进行，而非提交用户输入
    3. 不同上下文有不同转义规则，业务人员需选取合适的转义库

    # 12. 性能优化

    性能分析

    • 页面加载性能（加载时间、用户体验）
    • 动画与操作性能（是否流程无卡顿）
    • 内存占用（内存占用过大，浏览器崩掉）
    • 电量消耗（游戏方面）

    优化方案

    缓存：客户端控制的强缓存策略。

    降低请求成本：客户端控制，隔一段时间主动请求dns获取域名。服务器升级到http2。采用cdn管理大型文件、图片。

    减少请求：js、css打包到html中。js控制异步加载、懒加载。小型图片使用data-url。

    减少传输体积：尽量使用svg等代替图片。根据机型、网络控制图片清晰度。避免大型背景图。

    # 13. DOM

    dom：文档对象模型。在网页上，组织页面（或文档）的节点对象被组织在一个树形结构中，用来表示文档中对象的标准模型就称为 DOM。

    节点与元素：节点分为元素节点、文本节点(注释)、文档节点等，元素节点即由标签表示，如<html>，<head>，<title>，<body>，<h2>，<p>

    相关api：

    节点创造型：createElement创造元素、createTextNode创建文本节点、cloneNode克隆节点、createDocumentFragment

    节点修改型：appendChild、insertBefore、removeChild、replaceChild

    节点关系型：parentNode、parentElement、children、childNodes、firstChild、lastChild

    元素属性型：setAttribute、getAttribute、hasAttribute

    样式操作型：element.style.color = ...等等

    # 14. Ajax（异步请求技术）

    Ajax：是一种由xml+javascript组合起来的一种异步请求技术.可实现动态刷新。

    原理：简单来说通过XmlHttpRequest对象来向服务器发送异步请求，从服务器获得数据，然后用JavaScript来操作DOM而更新页面。这其中最关键的一步就是从服务器获得请求数据。要清楚这个过程和原理，我们必须对 XMLHttpRequest有所了解。

    与jquery的关系：Jquery是JavaScript的框架，而Ajax是JavaScript的一种应用。Jquery在Ajax的基础上进行了封装了，所以在Jquery中可以用Ajax。

    jquery中封装的ajax：

    $.ajax({
    method: "POST",
    url: "http://localhost/api",
    data: { name: "Adnan", country: "Iran" }
}).done(
    response => console.log(response))
.fail(
    () => console.log('error'));

    fetch（基于promise，Fetch 是一个 API，原生的js对象而不是XHR，它是真实存在的）

    fetch('http://localhost/api',{
    method: 'POST'
    body: { name: "Adnan", country: "Iran" }
      }).then(response => console.log(response));

    async function getData() {
    let response = await fetch('http://localhost/api' {
                               method: 'POST'
                               body: { name: "Adnan", country: "Iran" }
});
return response;
}

    axios（基于promise，基于 XHR 进行二次封装的网络请求库）

    Axios 可以说是 XHR 的一个子集，而 XHR 又是 Ajax 的一个子集

    axios({
    method: 'POST',
    url: 'http://localhost/api',
    data: { name: "Adnan", country: "Iran" }
}).then(response => console.log(response))
    .catch(err => console.log(err));

// axios也可以跟 async/await 结合使用：
async function getData() {
    let response = await axios.post('http://localhost/api' {
                                    name: "Adnan",
                                    country: "Iran"
                                    });
return response;
}

    # 15. Babel

    babel：js编译器，把使用ES6/ES7等“高级”语法编写的Js代码转换为ES5/ES3的“通俗”语法（也可以把JSX语法转为Javascript）。

    转译过程：

    1. parse解析：将代码字符串解析成抽象语法树（AST）=> @babel/parser 的 parse 方法
    2. transform转换：对AST进行处理，将高版本语法的AST转化为低版本语法的AST=> @babel/traverse
    3. generator生成：将转好的AST转为代码字符串 => @babel/generator
    4. 过程中，可能还需要使用 @babel/traverse 来获取依赖文件等

    # 16. webpack

    # 16.1 webpack作用、过程与loader与plugin

    作用：找到JavaScript模块以及其它的一些浏览器不能直接运行的拓展语言（Sass，TypeScript等），并将其转换和打包为合适的格式供浏览器使用。

    **打包过程：**通过定义 entry.js，对所有依赖的文件进行跟踪，将各个模块通过 loader （模块转换器，使webpack可以加载和解析非js代码）和 plugins （在webpack构建流程中的特定时机注入扩展逻辑）处理，然后打包在一起。(1)初始化参数 (2)开始编译 (3)确定入口entry (4)编译loader模块 (5) 打包输出资源 (6)输出完成

    功能：代码转换、文件优化、代码分割、模块合并、代码校验、自动发布

    • loader 运行在打包文件之前。对于loader，实质是一个转换器，将A文件进行编译形成B文件，操作的是文件，比如将A.scss或A.less转变为B.css，单纯的文件转换过程
    • plugins 在整个编译周期都起作用。在Webpack 运行的生命周期中会广播出许多事件，Plugin 可以监听这些事件，在合适的时机通过Webpack**提供的 API**改变输出结果

    常见loader：

    • css-loader读取 合并CSS 文件
    • style-loader把 CSS 内容注入到 JavaScript 里
    • sass-loader 解析sass文件（安装sass-loader，node-sass）
    • postcss-loader自动添加浏览器兼容前缀（postcss.config配置）
    • url-loader将文件转换为base64 URI。
    • vue-loader处理vue文件。

    常见plugin：

    • HtmlWebpackPlugin（打包后的静态文件自动插入到这个index.html 文件当中）
    • CleanWebpackPlugin（自动清理输出文件夹）
    • PurgeCSSPlugin（删除没有引用到的选择器及其样式）
    • CompressionPlugin（传输压缩，gzip）
    • HotModuleReplacementPlugin（热更新）

    **sourceMap：**是一个映射关系，将打包后的文件隐射到源代码，用于定位报错位置

    # 16.2 热更新hot

    热更新：当你对代码修改并保存后，将会对代码进行重新打包，并将改动的模块发送到浏览器端，浏览器用新的模块替换掉旧的模块，去实现局部更新页面而非整体刷新页面。

    # 16.3 如何利用webpack优化前端性能？

    答：即指优化webpack的输出结果，让打包的最终结果在浏览器运行快速高效。

    1. 压缩代码，删除多余的代码、注释、简化代码的写法等等方式。
    2. 利用cdn加速，将引用的静态资源路径修改为CDN上对应的路径。
    3. 删除死代码，js（tree shaking），css（purify-css）
    4. 提取公共代码。使用commonsChunkPlugin插件

    # 16.4 运行npm run xxx发生了什么？

    在执行npm run xxx的时候，会在package.json找到指令对应的命令。

    那为什么不直接执行对应的指令？

    因为系统里只有npm相关的命令。npm会在node_modules/.bin目录里找到对应的映射文件（软连接），然后再找到相对应的js文件来执行。

    # 17. 微前端

    定义：一种软件架构，将前端分为一小个子应用，每个子应用互相独立，可以分开开发，测试，部署，技术栈也可以不一样，从而兼顾老项目。这些子应用程序可以在同一个页面上共存，并相互之间可以进行通信和交互。

    实现：iframe，js框架如qiankun

    1. 样式隔离（基座与子应用）：css-module加前缀或统一样式

    2. 子应用间跳转：

      history.pushState()

      将主应用的路由实例通过 props 传给微应用，微应用这个路由实例跳转。

    3. 公共依赖：

      主应用：将所有公共依赖配置webpack 的externals，并且在index.html使用外链引入这些公共依赖

      子应用：和主应用一样配置webpack 的externals，并且在index.html使用外链引入这些公共依赖，注意，还需要给子应用的公共依赖的加上 ignore 属性（这是自定义的属性，非标准属性），qiankun在解析时如果发现igonre属性就会自动忽略

    4. 全局状态管理：qinkun提供了一个全局的GlobalState来共享数据，基座初始化之后，子应用可以监听到这个数据的变化，也能提交这个数据。

    # 18.1 路由懒加载

    react

    React.lazy()+Suspense

    const Home = lazy(() => import('./routes/Home'));

const App = () => (
  <Router>
    <Suspense fallback={<div>Loading...</div>}>

    vue

    // 将
// import UserDetails from './views/UserDetails'
// 替换成
const UserDetails = () => import('./views/UserDetails')

const router = createRouter({
  // ...
  routes: [{ path: '/users/:id', component: UserDetails }],
})

    原理

    因为懒加载是对子模块（子组件）进行延后加载。

    前提知识：通过webpack打包时会进行资源整合，也就是会把项目中的JS、CSS等文件按照一定的规则进行合并，已达到减少资源请求的目的。

    第一步就是将懒加载的子模块（子组件）分离出来。

    通过import()引用的子模块会被单独分离出来，打包成一个单独的文件（打包出来的文件被称为chunk ）。

    第二步使用分离出来的chunk

    将需要进行懒加载的子模块文件的引入语句（本文特指import()）放到一个函数内部。然后在需要加载的时候再执行该函数。

    webpack会对对应的chunk进行识别解析
